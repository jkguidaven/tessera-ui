## Project Overview

**Tessera UI** is a framework-agnostic web component library built with **Stencil.js** and **TypeScript**. Components are authored once and compiled to standards-compliant Web Components with automatic wrapper generation for React, Vue, and Angular.

## Coding Conventions

### Component Authoring Rules

1. **One component per directory** вЂ” each component gets its own folder under `src/components/`.
2. **File naming** вЂ” use kebab-case: `my-component.tsx`, `my-component.css`, `my-component.spec.ts`.
3. **Component tag prefix** вЂ” all components use the `ts-` prefix (e.g., `<ts-button>`, `<ts-card>`).
4. **Props** вЂ” use Stencil's `@Prop()` decorator. Always provide JSDoc comments. All props must have sensible defaults.
5. **Events** вЂ” use Stencil's `@Event()` decorator with the `q` prefix for custom events (e.g., `tsClick`, `tsChange`). Events must emit `CustomEvent` with typed detail.
6. **Slots** вЂ” prefer named slots for complex components. Always document slot names.
7. **CSS** вЂ” scope all styles to Shadow DOM. Use CSS custom properties (prefixed `--ts-`) for anything consumers should customize.
8. **No external CSS frameworks** вЂ” no Tailwind, Bootstrap, etc. All styling is authored from scratch using design tokens.
9. **Reference implementation** вЂ” when in doubt, follow the pattern in `src/components/button/` as the canonical example.

### TypeScript Rules

- Strict mode enabled (`"strict": true`).
- No `any` types вЂ” use `unknown` and narrow with type guards.
- Export all public types from `src/types/index.ts`.
- Prefer interfaces over type aliases for object shapes.
- All functions must have explicit return types (except test files).
- Use `import type { ... }` for type-only imports.

### Accessibility Requirements

- All interactive components must be keyboard navigable.
- ARIA attributes are mandatory where native semantics are insufficient.
- Color contrast must meet WCAG 2.1 AA (4.5:1 for text, 3:1 for large text/UI).
- Use `src/utils/aria.ts` helpers for consistent ARIA patterns (focus trapping, ID generation, screen reader announcements).
- Test keyboard navigation in E2E tests.

### Testing Requirements

- Every component must have a `.spec.ts` (unit) and `.e2e.ts` (integration) file.
- Unit tests cover: prop reflection, event emission, slot rendering, accessibility attributes.
- E2E tests cover: keyboard navigation, visual regression, cross-browser rendering.
- Minimum 80% code coverage.
- Test disabled/loading/error states вЂ” not just the happy path.

### Design Token Conventions

Tokens follow a **three-tier architecture** defined in `src/theme/tokens.css`:

#### Tier 1 вЂ” Reference Tokens (`--ts-ref-*`)

Raw palette values. **Never** used directly in component CSS.

```
--ts-ref-primary-500    # Raw primary hex
--ts-ref-neutral-300    # Raw neutral hex
```

#### Tier 2 вЂ” Semantic Tokens (`--ts-*`)

Role-based tokens that alias reference tokens. Components consume these.

```
--ts-color-text-primary          # Text color roles
--ts-color-bg-surface            # Background roles
--ts-color-border-default        # Border roles
--ts-color-interactive-primary   # Interactive state roles
--ts-color-info-bg               # Feedback background roles
--ts-color-primary-500           # Legacy palette aliases (backwards-compatible)
--ts-spacing-4                   # Spacing (4px base)
--ts-radius-md                   # Border radius
--ts-shadow-md                   # Elevation / shadows
--ts-transition-fast             # Motion tokens
```

#### Tier 3 вЂ” Component Tokens (`--ts-{component}-*`)

Per-component tokens defined on `:host`, defaulting to semantic tokens. Consumers override these to customize individual components.

```
--ts-button-radius        # Button border radius
--ts-input-border-color   # Input border color
--ts-card-bg              # Card background
--ts-modal-overlay-bg     # Modal overlay
--ts-tooltip-bg           # Tooltip background
```

#### Adding tokens for a new component

1. Define component tokens on `:host` in the component's `.css` file
2. Default each to a semantic token (Tier 2): `--ts-mycomp-bg: var(--ts-color-bg-surface);`
3. Use component tokens (not semantic tokens) in internal selectors
4. Document component tokens in a comment block at the top of the CSS file
5. Variant-specific selectors may use semantic tokens directly
6. **Never** use raw hex/px/rgba values вЂ” always reference a token

#### Dark mode

- Activated via `[data-theme="dark"]` on any ancestor
- Reference tokens (Tier 1) are **immutable** вЂ” dark mode remaps semantic tokens (Tier 2)
- Component tokens inherit dark values automatically through the `var()` chain
- All overrides live in the `[data-theme="dark"]` block in `tokens.css`

## Common Agent Tasks

### Prerequisites

- **Node 20+** is required вЂ” run `nvm use` before any commands (`.nvmrc` is configured).
- Use `pnpm` for all package operations (not npm or yarn).

### Add a new component

```bash
# 1. Create component directory and required files (four files вЂ” stories are auto-generated)
mkdir -p src/components/my-component
touch src/components/my-component/{my-component.tsx,my-component.css,my-component.spec.ts,my-component.e2e.ts}

# 2. Follow the pattern in src/components/button/ as a reference implementation
# 3. Export from src/index.ts
# 4. Use `import type { EventEmitter }` (not value import) to satisfy lint
# 5. Stories are auto-generated on build вЂ” do NOT create .stories.ts manually
```

### Verification checklist (MANDATORY before committing)

After any component creation or modification, run ALL of these and ensure they pass:

```bash
pnpm build                    # Verifies all output targets compile + generates stories
pnpm test                     # Unit tests
pnpm test.e2e                 # E2E tests
npx eslint src --ext .ts,.tsx # Must have 0 errors (warnings are acceptable)
```

**Do NOT commit if any check fails.** Fix the issue first.

### Run tests

```bash
pnpm test              # Run all unit tests
pnpm test.e2e          # Run all E2E tests
pnpm test -- --spec src/components/button/button.spec.ts  # Single component
```

### Build the library

```bash
pnpm build             # Full production build (all output targets)
pnpm build.watch       # Stencil rebuild on changes (no server)
```

### Development server

```bash
pnpm start             # Full build + Stencil watch + Storybook with hot-reload
pnpm storybook         # Storybook standalone (needs prior pnpm build)
```

### Auto-generated Storybook stories

- `scripts/storybook-output-target.js` generates `.stories.ts` on every Stencil build
- Files with `// @auto-generated` marker are overwritten each build
- To hand-write stories: remove the marker вЂ” the generator will skip that file
- Adding/removing `@Prop()` automatically updates stories on next build

### Generate framework wrappers

```bash
pnpm generate:react    # Generate React wrapper package
pnpm generate:vue      # Generate Vue wrapper package
pnpm generate:angular  # Generate Angular wrapper package
```

## Output Targets

Stencil compiles to multiple output targets (configured in `stencil.config.ts`):

| Target                 | Purpose                                     |
| ---------------------- | ------------------------------------------- |
| `dist`                 | ES modules + lazy loading for bundlers      |
| `dist-custom-elements` | Tree-shakeable individual component exports |
| `www`                  | Static site for development/testing         |
| `docs-json`            | Custom Elements Manifest (JSON)             |
| `storybook-stories`    | Auto-generated Storybook stories            |
| `react`                | First-class React bindings                  |
| `vue`                  | First-class Vue bindings                    |
| `angular`              | First-class Angular bindings                |

## Key Design Decisions

1. **Why Stencil over Lit?** вЂ” Automatic framework wrapper generation. Maintaining React/Vue/Angular bindings by hand is the #1 pain point in cross-framework libraries. Stencil solves this at the compiler level.
2. **Why Shadow DOM?** вЂ” True style encapsulation. Consumer styles can't accidentally break components. Theming is handled entirely through CSS custom properties.
3. **Why pnpm workspaces?** вЂ” Each framework wrapper is a separate publishable package. pnpm is fastest for monorepo workflows with strict dependency management.
4. **Why CSS custom properties for tokens?** вЂ” They cross the Shadow DOM boundary, making theming trivial for consumers. No build step required to customize themes.

## Gotchas & Warnings

- **Do NOT import framework-specific code into `src/components/`** вЂ” components must remain framework-agnostic.
- **Shadow DOM limitations** вЂ” `::slotted()` only targets direct children of a slot. Deep styling requires CSS custom properties.
- **SSR** вЂ” Stencil supports hydration for SSR. When adding components, ensure they degrade gracefully without JS (no blank white boxes).
- **Form participation** вЂ” use `ElementInternals` API for form-associated custom elements. See `src/components/input/` for the pattern.
- **Event bubbling** вЂ” Custom events from Shadow DOM do NOT bubble by default. Set `{ bubbles: true, composed: true }` on events that need to cross shadow boundaries.
- **No external runtime** вЂ” `dist-custom-elements` is configured with `externalRuntime: false`. Components are self-contained.
- **Naming collisions** вЂ” the `ts-` prefix avoids collisions with native HTML and other component libraries. Never remove it.
