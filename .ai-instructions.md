## Keeping Agent Instructions Up To Date

When any change affects the development workflow, tooling, scripts, conventions, or project structure, update this file (`.ai-instructions.md`) and `CLAUDE.md` in the same commit. These files are the source of truth for all AI agents — stale instructions cause repeated mistakes across all tools.

## Project Overview

**Tessera UI** is a framework-agnostic web component library built with **Stencil.js** and **TypeScript**. Components are authored once and compiled to standards-compliant Web Components with automatic wrapper generation for React, Vue, and Angular.

## Coding Conventions

### Component Authoring Rules

1. **One component per directory** — each component gets its own folder under `src/components/`.
2. **File naming** — use kebab-case: `my-component.tsx`, `my-component.css`, `my-component.spec.ts`.
3. **Component tag prefix** — all components use the `ts-` prefix (e.g., `<ts-button>`, `<ts-card>`).
4. **Props** — use Stencil's `@Prop()` decorator. Always provide JSDoc comments. All props must have sensible defaults.
5. **Events** — use Stencil's `@Event()` decorator with the `q` prefix for custom events (e.g., `tsClick`, `tsChange`). Events must emit `CustomEvent` with typed detail.
6. **Slots** — prefer named slots for complex components. Always document slot names.
7. **CSS** — scope all styles to Shadow DOM. Use CSS custom properties (prefixed `--ts-`) for anything consumers should customize.
8. **No external CSS frameworks** — no Tailwind, Bootstrap, etc. All styling is authored from scratch using design tokens.
9. **Reference implementation** — when in doubt, follow the pattern in `src/components/button/` as the canonical example.

### TypeScript Rules

- Strict mode enabled (`"strict": true`).
- No `any` types — use `unknown` and narrow with type guards.
- Export all public types from `src/types/index.ts`.
- Prefer interfaces over type aliases for object shapes.
- All functions must have explicit return types (except test files).
- Use `import type { ... }` for type-only imports.

### Accessibility Requirements

- All interactive components must be keyboard navigable.
- ARIA attributes are mandatory where native semantics are insufficient.
- Color contrast must meet WCAG 2.1 AA (4.5:1 for text, 3:1 for large text/UI).
- Use `src/utils/aria.ts` helpers for consistent ARIA patterns (focus trapping, ID generation, screen reader announcements).
- Test keyboard navigation in E2E tests.

### Testing Requirements

- Every component must have a `.spec.ts` (unit) and `.e2e.ts` (integration) file.
- Unit tests cover: prop reflection, event emission, slot rendering, accessibility attributes.
- E2E tests cover: keyboard navigation, visual regression, cross-browser rendering.
- Minimum 80% code coverage.
- Test disabled/loading/error states — not just the happy path.

### Design Token Conventions

Tokens follow a **three-tier architecture** defined in `src/theme/tokens.css`:

#### Tier 1 — Reference Tokens (`--ts-ref-*`)

Raw palette values. **Never** used directly in component CSS.

```
--ts-ref-primary-500    # Raw primary hex
--ts-ref-neutral-300    # Raw neutral hex
```

#### Tier 2 — Semantic Tokens (`--ts-*`)

Role-based tokens that alias reference tokens. Components consume these.

```
--ts-color-text-primary          # Text color roles
--ts-color-bg-surface            # Background roles
--ts-color-border-default        # Border roles
--ts-color-interactive-primary   # Interactive state roles
--ts-color-info-bg               # Feedback background roles
--ts-color-primary-500           # Legacy palette aliases (backwards-compatible)
--ts-spacing-4                   # Spacing (4px base)
--ts-radius-md                   # Border radius
--ts-shadow-md                   # Elevation / shadows
--ts-transition-fast             # Motion tokens
```

#### Tier 3 — Component Tokens (`--ts-{component}-*`)

Per-component tokens defined on `:host`, defaulting to semantic tokens. Consumers override these to customize individual components.

```
--ts-button-radius        # Button border radius
--ts-input-border-color   # Input border color
--ts-card-bg              # Card background
--ts-modal-overlay-bg     # Modal overlay
--ts-tooltip-bg           # Tooltip background
```

#### Adding tokens for a new component

1. Define component tokens on `:host` in the component's `.css` file
2. Default each to a semantic token (Tier 2): `--ts-mycomp-bg: var(--ts-color-bg-surface);`
3. Use shape family tokens for border-radius: `--ts-mycomp-radius: var(--ts-shape-interactive);`
4. Use component tokens (not semantic tokens) in internal selectors
5. Document component tokens in a comment block at the top of the CSS file
6. Variant-specific selectors may use semantic tokens directly
7. **Never** use raw hex/px/rgba values — always reference a token
8. Use CSS **logical properties** (not physical): `padding-inline-start` not `padding-left`

#### Global Configuration Pattern (data-* attributes)

All global configuration in Tessera UI uses **HTML `data-*` attributes on ancestor elements** — never JavaScript configuration objects or provider components. This ensures:
- Works in any framework (React, Vue, Angular, Svelte, plain HTML)
- No JavaScript required — pure HTML/CSS
- Scopable at any DOM level (page-wide on `<html>`, or per-section on any container)
- Consistent API surface for consumers

**Current `data-*` configuration attributes:**

| Attribute | Values | Default | Purpose |
|---|---|---|---|
| `data-theme` | `"dark"`, `"high-contrast"` | (light) | Color theme — remaps semantic tokens |
| `data-density` | `"compact"`, `"spacious"` | (comfortable) | Spacing/sizing density — remaps spacing and font-size tokens |
| `data-icons` | `"none"` | (Lucide built-in) | Icon resolution — disables built-in Lucide fallback |
| `dir` | `"rtl"` | `"ltr"` | Text direction — logical properties adapt automatically |

**Rules for adding new global options:**
1. Use a `data-*` attribute, not a JS API or provider component
2. The default (no attribute) should be the most common/batteries-included behavior
3. The attribute must be checkable via `this.el.closest('[data-X]')` inside components
4. Document the attribute in this table and in the relevant design language page
5. `@media` queries (like `forced-colors`, `prefers-reduced-motion`) are the exception — these are browser-controlled, not app-controlled
6. Component tokens inherit all theme/density/icon values automatically through the `var()` chain or ancestor lookup

#### Key token categories

- **Colors**: Primary (orange), Secondary (teal), Neutral (warm), Success, Warning, Danger, Info — all with full 50-900 ramps
- **On-color tokens**: `--ts-color-text-on-primary`, `--ts-color-text-on-success`, etc.
- **Container tokens**: `--ts-color-primary-container` / `--ts-color-on-primary-container` pairs
- **State layers**: `--ts-state-hover-opacity` (0.08), `--ts-state-focus-opacity` (0.12), etc.
- **Shape families**: `--ts-shape-interactive`, `--ts-shape-container`, `--ts-shape-overlay`, `--ts-shape-badge`
- **Type roles**: `--ts-type-display-lg-*`, `--ts-type-heading-md-*`, `--ts-type-body-md-*`, `--ts-type-label-sm-*`, `--ts-type-caption-*`
- **DTCG JSON**: Token source files in `tokens/` directory (W3C Design Tokens Community Group format)

## Common Agent Tasks

### Prerequisites

- **Node 20+** is required — run `nvm use` before any commands (`.nvmrc` is configured).
- Use `pnpm` for all package operations (not npm or yarn).

### Add a new component

```bash
# 1. Create component directory and required files (five files)
mkdir -p src/components/my-component
touch src/components/my-component/{my-component.tsx,my-component.css,my-component.spec.ts,my-component.e2e.ts,my-component.stories.ts}

# 2. Follow the pattern in src/components/button/ as a reference implementation
# 3. Export from src/index.ts
# 4. Use `import type { EventEmitter }` (not value import) to satisfy lint
# 5. Hand-write stories — do NOT use @auto-generated marker
# 6. Stories must showcase: Default, Variants, Sizes, States, Composition
# 7. Use realistic content and Lucide icon names in examples
```

### Verification checklist (MANDATORY before committing)

After any component creation or modification, run ALL of these and ensure they pass:

```bash
pnpm build                    # Verifies all output targets compile + generates stories
pnpm test                     # Unit tests
pnpm test.e2e                 # E2E tests
npx eslint src --ext .ts,.tsx # Must have 0 errors (warnings are acceptable)
```

**Do NOT commit if any check fails.** Fix the issue first.

### Run tests

```bash
pnpm test              # Run all unit tests
pnpm test.e2e          # Run all E2E tests
pnpm test -- --spec src/components/button/button.spec.ts  # Single component
```

### Build the library

```bash
pnpm build             # Full production build (all output targets)
pnpm build.watch       # Stencil rebuild on changes (no server)
```

### Development server

```bash
pnpm start             # Full build + Stencil watch + Storybook with hot-reload
pnpm storybook         # Storybook standalone (needs prior pnpm build)
```

### Storybook stories

All stories are **hand-written** — the auto-generator is disabled for all components.

- Stories must NOT contain `@auto-generated` (this triggers overwrite on build)
- When creating or updating a component, **always update its `.stories.ts`** to reflect changes
- Required story variants: **Default** (interactive controls), **Variants**, **Sizes**, **States** (disabled/error/loading), **Composition** (realistic multi-component usage)
- Use realistic content ("Save Changes", "john@example.com") — never "Content" or "Label"
- Use `<ts-icon name="...">` with Lucide icon names in examples
- For compound components (tabs + tab-panel), put stories in the parent file only. Create a stub for the child: `// Stories for this component are in the parent component's stories file`
- Reference: `src/components/button/button.stories.ts`

### Generate framework wrappers

```bash
pnpm generate:react    # Generate React wrapper package
pnpm generate:vue      # Generate Vue wrapper package
pnpm generate:angular  # Generate Angular wrapper package
```

| Target                 | Purpose                                     |
| ---------------------- | ------------------------------------------- |

| `dist-custom-elements` | Tree-shakeable individual component exports |
| `www`                  | Static site for development/testing         |
| `docs-json`            | Custom Elements Manifest (JSON)             |
| `storybook-stories`    | Auto-generated Storybook stories            |
| `react`                | First-class React bindings                  |
| `vue`                  | First-class Vue bindings                    |
| `angular`              | First-class Angular bindings                |

## Key Design Decisions

1. **Why Stencil over Lit?** — Automatic framework wrapper generation. Maintaining React/Vue/Angular bindings by hand is the #1 pain point in cross-framework libraries. Stencil solves this at the compiler level.
2. **Why Shadow DOM?** — True style encapsulation. Consumer styles can't accidentally break components. Theming is handled entirely through CSS custom properties.
3. **Why pnpm workspaces?** — Each framework wrapper is a separate publishable package. pnpm is fastest for monorepo workflows with strict dependency management.
4. **Why CSS custom properties for tokens?** — They cross the Shadow DOM boundary, making theming trivial for consumers. No build step required to customize themes.

## Gotchas & Warnings

- **Do NOT import framework-specific code into `src/components/`** — components must remain framework-agnostic.
- **Shadow DOM limitations** — `::slotted()` only targets direct children of a slot. Deep styling requires CSS custom properties.
- **SSR** — Stencil supports hydration for SSR. When adding components, ensure they degrade gracefully without JS (no blank white boxes).
- **Form participation** — use `ElementInternals` API for form-associated custom elements. See `src/components/input/` for the pattern.
- **Event bubbling** — Custom events from Shadow DOM do NOT bubble by default. Set `{ bubbles: true, composed: true }` on events that need to cross shadow boundaries.
- **No external runtime** — `dist-custom-elements` is configured with `externalRuntime: false`. Components are self-contained.
- **Naming collisions** — the `ts-` prefix avoids collisions with native HTML and other component libraries. Never remove it.
